Building a QR‑Based Secret Vault (And Why Google Authenticator Might Already Solve Your Problem)

We all reach that moment where secrets start piling up. Could you build your own secure QR‑based vault? Yes, you absolutely can.

We all reach that moment where passwords, API keys, recovery tokens, and “scan this QR to set things up” secrets start piling up. You want a safer way to store them, ideally something that works across devices, feels modern, and doesn’t leak sensitive data.

“Could I build my own secure QR‑based vault?”

The short answer is yes—you absolutely can. And I’ll walk you through what that looks like. But along the way, you might also discover that Google Authenticator (or similar apps) already solves a big part of the problem, especially if your goal is account safety rather than custom secret workflows.

1. Architecture Overview

graph TD A[QR Code Source] --> B[Camera / Photo Upload] B --> C[QR Decoder (html5-qrcode)] C --> D[Flask API Layer] D --> E[Fernet Encryption (AES-GCM)] E --> F[SQLite Storage] F --> G[UI Masked Preview]

This design ensures secrets are encrypted at rest, and the UI never displays full values. Scanning works via camera or photo upload, all running in a single Python file.

2. Algorithm Composition

The system is built around three core logic transformations:

  • Ingestion Logic: The Html5Qrcode library handles the client-side heavy lifting, decoding the image buffer into a raw UTF-8 string before transmitting it to the backend.
  • Encryption Logic: On receipt, the server derives a secure key from your environment's MASTER_SECRET. Using **Fernet (AES-GCM)**, the raw secret is transformed into an opaque blob.
  • Entities Composition: The system stores the entity as a record containing the blob, the auto-detected content type (URL, JSON, or Secret), and a timestamp. When queried, it recovers the secret but only exposes a masked preview to the UI.

3. Code Structure Output

The project follows a "Single-File Microservice" pattern, ideal for local deployment or containerization:

qr_vault/
├── app.py # Main Flask + PWA Logic
├── requirements.txt # flask, cryptography
├── README.md # Security/Setup Guide
└── qr_vault.db # Encrypted SQLite Storage (Auto-created)

Custom Vault vs. Google Authenticator

While building a custom vault is exciting, Google Authenticator already solves the core problem for most users. If your primary need is Protecting your accounts via TOTP seeds, stick with the battle-tested apps.

However, choose the Custom QR Vault when you need to store non-TOTP secrets like API keys, SSH tokens, or database passwords, and you want full control over where your data lives.
